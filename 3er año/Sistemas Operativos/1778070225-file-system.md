# File System

## ¿Qué es un File System?

Un **file system** permite a los usuarios organizar sus datos para que se persistan a través de un largo período de tiempo.

Formalmente un file system es:

> [!Quote] Una abstracción del sistema operativo que provee datos persistentes con un nombre

El hecho de que los datos tengan un nombre es con la intención de que podamos acceder a ellos por un identificador que el file system le asocia al dato.

## ¿Qué es un archivo?

Es una colección de datos con un nombre específico.

Proveen una abstracción de más alto nivel que la que subyace en el dispositivo de almacenamiento; un archivo proporciona un nombre único y con significado para referirse a una cantidad arbitraria de datos.

### Características
- **Metadata**: Información acerca del archivo que es comprendida por el SO
	- Tamaño
	- Fecha de modificación
	- Propietario
	- Información de seguridad
- **Datos**

Un archivo se implementa como un **dentry** apuntando a un **inodo**.

### Inodo
Un inodo (index node) almacena información sobre un archivo.

Un inodo contiene metadatos del archivo, pero **no el contenido del archivo en sí**.

![[Pasted image 20260506110103.png]]

#### Tipos de inodo
Los tipos de inodo en Linux nos dan idea de la variedad de "objetos" que podemos anclar al file system.

Se especifican en el campo **mode**.

### Dentry
Un Dentry (directory entry) representa la relación entre el nombre de un archivo y su inodo correspondiente.

**Los directorios son listas de dentries**.

La función principal de un dentry es ayudar a resolver nombres de archivos y rutas en el sistema de archivos. Cuando el sistema de archivos busca un archivo, utiliza las dentries para traducir la ruta al inodo correspondiente, que contiene la información del archivo.

Puede haber más de un Dentry apuntando a un mismo inodo. Esto es efectivamente un *hard link*

#### Directorios
**Son archivos especiales de tipo "directorio"**. El contenido de estos archivos es una lista de dentries.
- Un dentry no apunta a otro dentry, solo a inodos.
- Un inodo no apunta a otros inodos, solo a datos.
- Pero un dentry puede apuntar a un inodo tipo directorio, que apunta a datos y los datos contienen una lista de dentries. Así se implementa el sistema jerárquico!

![[Pasted image 20260506111542.png]]
PD: Un directorio es tratado como un archivo normal.

## Virutal File System

El virtual file system es el subsistema del kernel que implementa la interfaz que tiene que ver con los archivos y el sistema de archivos provistos a los programas corriendo en modo usuario. Todos los sistemas de archivos deben basarse en VFS para:
- Coexistir
- Inter-operar
Esto habilita a los programas a usar las syscalls para leer y escribir en diferentes sistemas de archivos y diferentes medios.

![[Pasted image 20260506113434.png]]

VFS es el pegamento que habilita a las syscalls a funcionar sin que estas necesiten tener en cuenta el hardware subyacente.

### File System Abstraction Layer
Esta capa de abstracción trabaja mediante la definición de interfaces conceptualmente básicas y de estructuras que cualquier sistema de archivos soporta.
Los file systems amoldan su visión de conceptos como "esta es la forma de cómo abro n archivo" para matchear las expectativas del VFS, todos estos sistemas de archivos soportan nociones tales como archivos, directorios y además todos soportan un conjunto de operaciones básicas sobre estos.
El resultado es una capa de abstracción general que le permite al kernel manejar muchos tipos de sistemas de archivos de forma fácil y limpia.

![[Pasted image 20260506114036.png]]

### VFS: Sus objetos
VFS presenta una serie de estructuras que modelan un file system, estas estructuras se denominan *objetos* (programadas en C). Estos objetos son:
- El **super bloque**, que representa a un sistema de archivos.
- El **inodo**, que representa a un determinado archivo.
- El **dentry**, que representa una entrada de directorio, que es un componente simple de un path.
- El **file** que representa a un archivo asociado a un determinado proceso.

## Dispositivos

### Estructura para el acceso a dispositivos
![[Pasted image 20260506114746.png]]

- Los dispositivos se representan como archivos especiales de tipo DEVICE
- Se usan las mismas syscalls que para operar con archivos normales
- El VFS interactua con una capa de Drivers
- Los Drivers son los que implementan las operaciones de bajo nivel que interactuan con el hardware.

### Tipos de dispositivos
Existen dos tipos fundamentales de dispositivos en Unix que se pueden vincular como nodos al file system.
- [[Dispositivos de bloque]]
- [[Dispositivos de caracter]]

### Cómo se vinculan los dispositivos al file system
Los dispositivos de I/O tienen un par de números asociados llamados números **major** y **minor**.
**Estos número implementan un sistema muy básico de ruteo hacia los dispositivos en sí**

### Cómo conectan los dispositivos al file system
Los dispositivos no son más que archivos especiales de **tipo DEVICE** que se crean usando la syscall `mknod`
