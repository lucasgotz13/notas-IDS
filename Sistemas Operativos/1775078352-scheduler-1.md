## Transición User-Kernel en xv6

> [!Info]- Conceptos clave
> Cambio de contexto, memoria virtual, trampoline, trapframe

### Estructura de un proceso en xv6
En la clase anterior se vio que un proceso tiene asociado un contexto. El contexto informalmente se puede pensar como el estado completo del proceso en ejecución conformado por:
- Espacio de direcciones de memoria
- Registros del procesador
- Estructuras del Kernel

Cuando el SO interrumpe momentáneamente la ejecución de un proceso y restaura la ejecución de uno suspendido se produce un **cambio de contexto**.

Para pasar de un proceso A a un proceso B, el sistema operativo pide un cambio de contexto.

El planificador/scheduler hace dos cosas:
- Se sigue ejecutando lo que estaba.
- Se pasa a ejecutar otro proceso.

El kernel guarda su contexto en algún lado y cargar el contexto del proceso B.

Un pasaje de modo kernel a modo usuario sucede cuando hay un cambio de contexto por ejemplo.

### Cambio de contexto: Resumiendo
1. Comienza con una transición a Kernel-Space.
2. Si se decide que hay que realizar un cambio de contexto, se invoca al scheduler.
3. El scheduler elije el siguiente proceso a ejecutar y llama a switch.
4. Switch le saca una foto al estado del kernel-space y del user-space (en general este ultimo ya estaba guardado desde 1) y restaura el proceso candidato (que queda en Kernel-Space)
5. Regresa al User-Space del nuevo proceso

> [!Info] Nunca puede haber un cambio de contexto entre dos procesos **simultáneos**

### Comparativa xv6 y Linux pre-KPTI

![[Pasted image 20260405190328.png]]

- El *trapframe* es un espacio de memoria de 4096 bytes para guardar la foto del procesador cuando estaba ejecutándose antes del pasaje del modo usuario al modo kernel.
- El *trampoline* es parte del kernel. Manda el estado del proceso del usuario al trapframe y carga el estado del kernel. Es donde se hace el context switch.
#### Trampoline
- Código assembler para transicionar del modo usuario a modo kernel
- Conceptualmente es la única página del kernel que está todo el tiempo cargada en el user-space del proceso. Es la única página compartida por el address space del usuario y el address space del kernel.
- El resto del kernel se cargará cuando sea necesario (en un momento el kernel residía completo en el address space del proceso).
- Linux modernos usan *Kernel page-table isolation (KPTI)*, que es una técnica similar a la de xv6.
#### Trapframe
- Es el lugar donde el kernel guarda "la foto de los registros", para poder restaurarlos luego.
- Es un espacio donde el Kernel va a guardar el estado de todos los registros del procesador, previo al pasar al modo kernel, en el cual no hay garantías de que estos registros se hubieran conservado.
- Recíprocamente, el Kernel cuando restaure un proceso va a leer todos los registros guardados en el Trapframe y los colocara nuevamente en el procesador.

### Transición User-Kernel
La transición inicia cuando se genera una interrupción por software mediante ecall.
El procesador "salta" al **trampoline** y cambia automáticamente a Kernel Mode.
Código del trampoline:
1.  Guarda todos los registros del procesador en el **trapframe**. 
2. Cambia el registro *satp*, cambiando la tabla de páginas y efectivamente, el address space.
3. Indica en el registro sp apuntando al kstack del proceso actual.
4. Invoca la función *usertrap()*.

Luego para volver:
1. Invoca la función userret() en el trampoline.
2. Cambia el registro satp, cambiando la tabla de páginas y efectivamente, el address space.
3. Restaura todos los registros guardados en el trapframe.
4. Invoca sret para volver al punto original.

## Scheduling

> [!INFO]- Conceptos clave
> Métricas de planificación, FIFO, Shortest Job First (SJF), Shortest Time-to-completion (STCF), Round Robin (RR)

### Multiprogramación
¿Que es? => Existen muchos programas cargados en memoria preparados para ser ejecutados.

Multiples procesos están listos para ser ejecutados un determinado tiempo según el S.O. lo decidieses en base a ciertas políticas de planificación o scheduling.

### Time Sharing
Se refiere a compartir de forma concurrente un recurso computacional entre muchos usuarios por medio de las tecnologías de multiprogramación y la inclusión de interrupciones de reloj por parte del S.O., permitiendo a este último acotar el tiempo de respuesta del computador y limitar el uso de la CPU por parte de un proceso dado.



Cuando hay múltiples cosas que hacer, ¿Cómo se elige cuál de ellas hacer primero?
Debe existir algún mecanismo que permita determinar cuanto tiempo de CPU le toca a cada proceso. Ese período de tiempo que el kernel le otorga a un proceso se denomina **time slice o time quantum**.

### Workload
El Workload es la carga de trabajo de un proceso corriendo en el sistema. Es fundamental para determinar partes de las políticas de planificación.
Mejor el cálculo => mejor la planificación.

### Clasificación de schedulers

Tipo de sistema:
- **Interactivo**: necesita tiempos de respuesta bajos. Los procesos deben recibir atención frecuentemente. Ejemplo: Un procesador de texto.
- **Por lotes (batch)**: Los procesos pueden tardar más en completarse. Se optimiza el tiempo total de ejecución más que la respuesta inmediata. Ejemplo: FIFO, SJF, Calcular una liquidación de sueldos.
Preemptivo vs No Preemptivo:
- **Preemptivo**: El sistema puede interrumpir un proceso en ejecución para darle tiempo a otro. Permite mas **equidad** y **respuesta rápida**.
- **No Preemptivo**: Una vez que un proceso empieza, no se interrumpe hasta que termina (tipo FIFO y SJF).


> [!Info]-
> Para fines didácticos, se asumen las siguientes simplificaciones:
> - Cada proceso se ejecuta la misma cantidad de tiempo.
> - Todos los jobs llegan al mismo tiempo para ser ejecutados.
> - Una vez que empieza un job sigue hasta completarse.
> - Todos los jobs usan únicamente cpu (no realizan I/O).
> - El tiempo de ejecución (run-time) de cada job es conocido.

### Métricas de Planificación
Vamos a utilizar *turnaround time*. => Se define como el **"cual rápido se hacen las tareas"** (mide performance).
$$
T_{turnaround} = T_{completion} - T_{arrival}
$$
### Políticas de Scheduling
- **Batch**
	- FIFO
	- Shortest Job First (SJF)
	- Shortest Time-to-Completion (STCF)
- **Interactiva**
	- Round Robin (RR)
#### FIFO
El algoritmo mas básico para implementar.
Ventajas:
- Simple
- Funciona bárbaro para las suposiciones iniciales.
Desventajas:
- Si hay procesos que son muchos mas grandes que el otro, va a causar el *convoy effect* (los procesos mas chicos son afectados por el grande).

#### SJF
Para resolver el problema que presenta FIFO, se modifica la política para que se ejecute el proceso de duración mínima. 
Ventajas:
- Mejora el turnaround time para casos del convoy effect.
Desventajas:
- Si no todos los procesos llegan al mismo tiempo, el turnaround time incrementa demasiado.

#### STCF
Para poder solucionar el problema de SJF se necesita relajar la suposición 3. La idea es que el scheduler pueda adelantarse y determinar qué proceso debe ser ejecutado. Cuando determinados procesos "llegan", se pueden **desalojar (preempt)** al proceso A y decidir que otro proceso se ejecute y luego retomar.

### Tiempo de respuesta
La métrica del tiempo de respuesta surge con el advenimiento del time-sharing ya que los usuarios se sientan en una terminal de una computadora y pretenden una interacción con rapidez. => Nace el response time:
$$
T_{response} = T_{firstrun} - T_{arrival}
$$

¿Cómo escribimos un planificador que tenga noción del tiempo de respuesta?

### Round Robin
La idea del algoritmo es bastante simple:
- Se ejecuta un proceso por un período determinado de tiempo (**time slice**).
- Transcurrido el período se pasa a otro proceso, y así sucesivamente.
- Se lo conoce también como **time-slicing**.

Es muy importante la elección de un buen time slice, debido a que tiene que amortizar el cambio de contexto sin hacer que esto resulte en que el sistema no responda más.

## Multi-Level Feedback Queue

> [!INFO]- Conceptos clave
> Problemas que resuelve, Algoritmo MLFQ

Intenta atacar dos problemas:
- Optimizar el turnaround time => Lo logra ejecutando primero los tiempos más cortos, pero el SO generalmente no sabe cuánto tiempo va a durar un trabajo, el cuál es conocimiento requerido por SJF o STCF
- MLFQ intenta que el sistema sea responsivo a los usuarios interactivos **minimizando el response time**. Sin embargo, algoritmos como RR son buenos para reducir esto, pero terribles para el turnaround time.

### Las reglas básicas
MLFQ tiene un **conjunto de distinas colas**, c/u asignado un nivel de prioridad.

- **REGLA 1**: Si la prioridad (A) es mayor que la prioridad de (B), (A) se ejecuta y (B) no.
- **REGLA 2**: Si la prioridad de (A) es igual a la prioridad de (B), (A), y (B) se ejecutan en Round-Robin.

La clave subyace entonces en cómo el planificador setea las prioridades. **MLFQ** varía la prioridad de la tarea basándose en su comportamiento observado.

Mas uso de CPU => MLFQ asigna menos prioridad.
MLFQ va a aprender mientras los procesos se ejecutan para predecir su futuro comportamiento.

### ¿Cómo cambiar la prioridad?

Se debe decidir cómo MLFQ va a cambiar el nivel de prioridad a una tarea durante toda la vida de la misma.
Para hacer esto hay que tener en cuenta nuestro workload: una mezcla de tareas interactivas que tienen un corto tiempo de ejecución y que pueden renunciar a la utilización de la CPU y algunas tareas de larga ejecución basadas en la CPU que necesitan tiempo de CPU, pero poco tiempo de respuesta.

#### Mas reglas
- **REGLA 3**: Cuando llega una tarea al sistema se la coloca en la cola de más alta prioridad.
- **REGLA 4a**: Si durante una ejecución una tarea usa su time slice completo entonces su prioridad baja una unidad.
- **REGLA 4b**: Si una tarea renuncia al uso de la CPU antes de completar un time slice se queda en el mismo nivel de prioridad.

#### Problema con este approach
**Starvation**: Si hay demasiadas tareas interactivas en el sistema se van a combinar para consumir todo el tiempo del CPU y las tareas de larga duración nunca se van a ejecutar.
Se podría hackear el scheduler y obtener mas tiempo de CPU para determinados programas, lo cuál puede monopolizar todo el tiempo del CPU.

#### Segundo approach
Para cambiar el problema de starvation y permitir que las tareas con larga utilización de CPU puedan progresar lo que se hace es aplicar una idea simple, se mejora la prioridad de todas las tareas en el sistema. Se agrega una nueva regla:
- **REGLA 5**: Después de cierto periodo de tiempo S, se mueven las tareas a la cola con más prioridad.
De esta forma se previene el starvation.

Hay otro problema: prevenir que ventajeen al planificador.
Solución: Llevar una mejor contabilidad del tiempo de uso de la CPU en todos los niveles de MLFQ.

En lugar de que el planificador se **olvide** de cuanto time slice un determinado proceso utiliza en un determinado nivel el planificador debe hacer un seguimiento desde que un proceso ha sido asignado a una cola hasta que es trasladado a una cola de distinta prioridad.

**Cada proceso tiene un contador, que trackea cuanto time slice van consumiendo en total**.
